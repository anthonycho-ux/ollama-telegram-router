# ollama-telegram-router

*내가 만든 텔레그램 봇이 로컬 LLM으로 메시지를 라우팅하는 내용입니다. 그리고 지난 3주 동안 내가 이 방식을 잘못하고 있었다는 것을 증명하는 시간을 보냈습니다.*

## 추측했던 나의 버전

한동안, 라우터는 이런 방식으로 작동했습니다:

```javascript
function routeModel(message) {
  if (TOOL_KEYWORDS.some(kw => message.toLowerCase().includes(kw)))
    return "qwen3.5:4b";
  if (QUALITY_KEYWORDS.some(kw => message.toLowerCase().includes(kw)))
    return "gemma4:e4b";
  return "qwen3.5:4b";
}
```

이건 똑똑해 보였습니다. 체계적이라는 느낌이 들었습니다. 나는 충분히 읽었기에, 복잡한 작업에는 큰 모델이 더 좋고 간단한 작업에는 작은 모델이 더 빠르다는 것을 알고 있었습니다. 키워드는 단지 구현 방식일 뿐이었습니다.

문제는 논리가 아니었습니다. 문제는 그 논리에 기억이 없었다는 것이었습니다.

## Karpathy가 가르쳐준 것들로 구축한 것

Andrej Karpathy의 자기회귀 모델 작업 접근 방식은 원칙적으로 간단합니다: **빠르게 구축하고, 측정하고, 배우고, 반복하는 것.** 아키텍처에 대해 더 깊이 생각할 필요도, 더 많은 논문을 읽을 필요도 없습니다. 실험을 실행하고, 데이터가 내가 애착을 가질 겨를도 없이 나의 믿음을 수정하도록 내버려 두는 것입니다.

이것이 내가 여기서 시도한 것입니다.

나는 벤치마크 루프를 구축했습니다 — 30개의 프롬프트, 두 개의 모델, 5분마다. 자율적이고, 자가 치유적이며, 누군가 돌봐줄 필요가 없었습니다. 나는 결론부터 시작하지 않았습니다. 직관만으로는 해결할 수 없는 불일치에서 시작했고, 그것을 정리해 줄 무언가를 구축했습니다.

프롬프트들은 합성 벤치마크가 아니었습니다. 그것들은 내 텔레그램 봇이 실제로 수행하는 것들, 즉 bash 명령어, 파일 읽기, 파일 쓰기, 디렉토리 목록 보기였습니다. 만약 모델이 그것들을 신뢰성 있게 처리할 수 있다면, 나는 어떤 모델인지, 그리고 왜 그런지 알고 싶었습니다. 만약 할 수 없다면, 그것 역시 알고 싶었습니다.

## 실험

2026년 4월, 나는 qwen3.5:4b와 gemma4:e2b에 대해 동일한 30개 프롬프트를 5분마다, 총 71 사이클 동안 쉬지 않고 실행했습니다. 방해받지 않고, 누군가 돌보지 않았습니다.

장비: NVIDIA RTX 3090 (24GB VRAM, 벤치마크 중 약 15GB 여유), AMD Ryzen 7 5700G, 16 코어. VRAM 한계는 현실입니다 — 세 모델 스택의 총 26GB에서, 메모리에 동시에 들어가지 않아 실행할 수 없었던 실험들이 있었습니다.

71 사이클 × 30 프롬프트 × 2 모델 = 42,600개의 개별 도구 호출 측정.

나는 무엇을 찾고 있는지 몰랐습니다. 단지 추측을 멈추고 싶다는 것만 알았습니다.

## 내가 잘못 생각했던 것들

데이터를 보기 전, 나는 가정했습니다: *큰 모델 = 복잡한 작업에 더 좋음 = 지연 시간 증가를 감수할 가치가 있음.*

71 사이클 이후:

| | qwen3.5:4b | gemma4:e2b |
|---|---|---|
| Mean accuracy | 98.6% | 98.7% |
| Perfect runs (30/30) | 46.5% | **66.2%** |
| Worst floor | 28/30 | 28/30 |
| Latency (warm) | **1.0s** | 1.2s |
| Combined score | **1.93** | 1.63 |

qwen3.5:4b가 종합 점수에서 이겼습니다. 하지만 gemma4:e2b는 qwen3.5:4b의 46% 대비 66%의 실행에서 30/30을 달성했습니다.

나는 이것을 예상하지 못했습니다. 더 빠른 모델이 일관성이 떨어지는 모델이기도 했습니다.

더 놀라운 것은: **두 모델이 어떤 것이 더 좋은지에 대해 31%의 사이클에서 의견이 달랐다는 것입니다.** 의견이 달랐을 때, 거의 항상 qwen3.5:4b가 성능이 떨어졌습니다 (31개 중 22회). gemma4:e2b는 단 5회만 성능이 떨어졌습니다. 그리고 두 모델이 동시에 성능이 떨어진 8개 사이클 — 가장 어려운 실패 — 에서는 그들이 같은 프롬프트에서 바닥을 찍는 적이 없었습니다. 실패는 겹치지 않고 보완적이었습니다.

데이터는 단순히 라우터만 수정한 것이 아니었습니다. 내가 가지고 있다는 것을 깨닫지 못했던 믿음 — 즉, 복잡해 보이는 모든 것에 대해 느린 모델로 라우팅하는 것이 안전한 선택이라는 믿음 — 을 수정했습니다.

## 모든 것을 바꾼 세 가지 숫자

### 1. temperature = 0.0

첫 번째 온도 연구는 거의 일어나지 않을 뻔했습니다. 그것은 10개 프롬프트의 측면 실행이었고, 빠르고 대충 진행되었습니다.

temperature 0.1의 qwen3.5:4b: 28/30. 내가 프로덕션에서 사용하던 온도와 동일했습니다.

동일 모델, 동일 프롬프트, temperature를 0.0으로 설정: **10개 프롬프트 세트에서 10/10.**

하나의 설정. 모델 변경 없음. 프롬프트 엔지니어링 없음. 나는 71 사이클 로그를 다시 살펴보고 qwen이 28/30을 기록한 8개 사이클을 발견했습니다. 이 각각은 진정한 모델 실패라기보다는 온도 아티팩트였을 가능성이 높았습니다.

프로덕션에 적용: 한 줄의 변경. 종합 점수: ~1.97.

### 2. OLLAMA_KEEP_ALIVE = -1

실험 루프에는 콜드 스타트 문제가 있었습니다. 모델이 디스크에서 로드될 때마다, 첫 결과가 나오기까지 50~60초가 걸렸습니다. 지속적으로 실행한다면, 71 사이클은 워밍업만으로 60시간 이상이 걸렸을 것입니다.

그러다 나는 `OLLAMA_KEEP_ALIVE=-1`을 발견했습니다. 하나의 환경 변수. 모델들이 사이클 사이에 VRAM에 뜨겁게 유지되었습니다.

콜드 스타트 시간이 50초에서 1초 미만으로 떨어졌습니다.

합리적인 시간 안에 실행할 수 없었던 루프가 이제 6시간 만에 실행되었습니다. 이것이 없었다면, 나는 아마 10 사이클 정도만 실행했을 것이고, 노이즈가 많은 초기 데이터만 보고 잘못된 결론을 내렸을 것입니다. 전체 실험은 이 하나의 변수에 달려 있었습니다.

### 3. NUM_PARALLEL = 8

튜닝 전, 단일 프롬프트 지연 시간: 2.11초.

`OLLAMA_NUM_PARALLEL=8` 설정 후: 0.86초. **2.4배 빠름.**

벤치마크 루프는 성능 최적화 루프이기도 했습니다. 나는 전체 사이클 시간을 절반 이상 줄이는 두 가지 설정을 발견했습니다.

## 나 자신에 대해 배운 것

키워드 라우터는 기술적으로 제한적일 뿐만 아니라, 편안했습니다. 그것이 내린 모든 추측은 결정처럼 보였습니다. 나는 대화에서 그 어떤 것도 방어할 수 있었습니다: *"복잡한 작업에는 gemma를 사용하도록 라우팅합니다. 왜냐하면..."*

실험 루프는 덜 편안했습니다. 그것은 점수를 기록했습니다. 매 5분마다 내가 측정해 달라고 요청하지 않은 진실한 무언가를 나에게 말해주었습니다.

71 사이클에 걸쳐, 신뢰 구간이 좁혀졌습니다. 0으로 좁혀지지는 않았습니다 — gemma4:e4b는 여전히 전체 벤치마크를 실행 중입니다 (50+ 사이클 진행 중), 그리고 COMPLEX → e4b 라우팅은 여전히 잠정적입니다. 하지만 qwen3.5:4b와 gemma4:e2b 사이의 비교는 이제 라우팅하기에 충분히 깨끗해졌습니다.

루프는 여전히 실행 중입니다. 매 5분마다 두 모델의 점수를 매기고 내가 읽을 수 있는 곳에 결과를 기록합니다. 나는 더 이상 라우팅 결정을 내리지 않습니다. 나는 루프가 라우팅이 무엇이어야 하는지 말하도록 내버려 둡니다.

## 작동하지 않은 것들 (그리고 그것이 여기에 있는 이유)

나는 최종 시스템에 포함되지 않은 여러 가지를 시도했고, 데이터가 그것들이 작동하지 않았다는 이유를 알려줍니다:

**플래너/호출자 분해.** Multi-LLM Agent와 ACAR 논문 모두 계획(planning)과 실행(execution)을 분리하는 것이 추가적인 모델 패스를 추가한다는 것을 보여줍니다. 99%의 정확도에서, 지연 시간 비용이 정확도 이점을 상쇄합니다. 나는 이것을 위해 논문을 읽을 필요가 없었습니다 — 종합 점수 계산이 나에게 같은 것을 말해주었습니다.

**키워드 정제.** 더 많은 키워드를 추가하거나, 더 세분화된 키워드를 추가하는 것은 근본적인 문제를 해결하지 못했습니다: 키워드는 의도 자체를 포착하는 것이 아니라, 의도의 *외양*을 포착하고 있었습니다. 스프레드시트 맥락에서의 "계산"과 금융 맥락에서의 "계산"은 동일한 단어와 반대되는 라우팅 결정을 요구합니다. 키워드가 많아질수록 잘못된 확신만 커졌습니다.

**3-모델 스택의 gemma4:e4b.** e4b는 프록시 연구에서 e2b보다 26% 더 높은 점수를 기록했습니다 (10개 프롬프트 세트에서 8/10 대 6/10). 그러나 VRAM 계산은 빠듯합니다: gemma4-26b (17GB) + e4b (9.6GB) + qwen3.5:4b (3.4GB) = 30GB. 나는 26GB를 가지고 있습니다. e4b의 전체 벤치마크는 여전히 누적 중입니다 — 깨끗해지면, qwen을 제거할지, gemma4-26b를 제거할지, 아니면 순차적 로딩으로 인한 지연 시간 증가를 감수할지 알게 될 것입니다.

가장 흥미로운 열린 질문은 이것입니다: 만약 라우팅 결정 자체가 쿼리별로 적응적일 수 있다면 어떨까요? ACAR 논문은 σ-probe 접근 방식을 설명합니다 — 각 메시지를 두 모델 모두에 통과시키고, 응답 분산을 측정하여 그에 따라 라우팅하는 것입니다. 그것이 다음 실험입니다. 그것은 자체 벤치마크 루프가 필요합니다. 지금 내가 실행하는 루프가 그 실험을 읽을 수 있게 만듭니다.

## 라우팅 로직

```
classify(message)
  TOOL_USE  → keyword: search, calculate, fetch, execute,
                        read file, write file, bash, git, curl...
  COMPLEX   → keyword: explain, analyze, architecture,
                        refactor, strategy, thorough, vs, versus...
  CASUAL    → short message (≤8 words), no tool/complex signals
  ROUTINE   → otherwise

route(message)
  TOOL_USE  → qwen3.5:4b   (98.6% accuracy, 1.0s, combined 1.93)
  CASUAL    → qwen3.5:4b   (fastest, no accuracy penalty)
  COMPLEX   → gemma4:e2b   (proxy for e4b — full benchmark accumulating)
  ROUTINE   → qwen3.5:4b   (default: fastest model)
```

COMPLEX → gemma 라우팅은 잠정적입니다. gemma4:e4b가 목표입니다 (프록시 연구에서 26% 더 높은 종합 점수, 더 많은 VRAM, 더 나은 추론). 하지만 전체 30개 프롬프트 벤치마크는 여전히 실행 중입니다. 데이터가 들어오면, 라우팅은 자동으로 업데이트됩니다.

## 나만의 실험 실행하기

`router.js`를 텔레그램 브리지에 넣으세요:

```javascript
const { routeMessage } = require('./router');

function routeModel(message) {
  const decision = routeMessage(message);
  // decision.model      → "qwen3.5:4b" or "gemma4:e4b"
  // decision.category   → "TOOL_USE" | "CASUAL" | "COMPLEX" | "ROUTINE"
  // decision.reasoning  → cites the cycle count and metric behind the decision
  return decision.model;
}
```

실험 루프를 시작하세요:

```bash
cd scripts && ./run_experiments.sh
# 5분 간격 자율 사이클, 자가 치유, ../benchmark-data/ 에 로그 기록
```
